## [v0.5.0] - 2026-04-28

### 新增功能 (Added)
* 新增 encode follow-only 能力：`libs/audio_core` / `libs/audio_api` 现在支持在不生成 PCM 的前提下单独构建 encode follow 数据，新增 `BuildEncodeFollowData(...)`、`bag_build_encode_follow_data(...)` 与 `bag_peek_encode_text_job_result_layout(...)`，让 Android 等前端可以先拿到 token / 歌词行 / raw payload timeline 布局，再决定是否继续取完整音频结果。
* 新增 `bag_encode_result_layout` 与对应 async job 结果布局查询，正式暴露 encode 结果中 PCM、hex/bin 文本、payload timeline、token timeline、歌词行与 line range 的容量信息，便于前端先分配目标 buffer，再按需提取结构化结果。
* 新增 `libs/audio_io` 的 `WBAG` metadata v6 写入与解析能力，`audio_io_metadata` / `FlipBitsAudioMetadata` 现可携带 `sample_rate_hz`、`payload_byte_count`、`input_source_kind`、`segment_count` 与 `segment_sample_counts`，让保存后的音频文件能够独立描述输入来源和长文本切段边界。

### 技术改进/重构 (Changed/Refactor)
* 统一 `libs/audio_core` 的 follow tokenization 规则：标点符号现在跨拉丁/CJK脚本都独立成 token，歌词行仍保留原始标点位置，且 CJK 文本继续按单字符切分，避免前端再自行拆分 `word+punctuation` 才能做 token 跟随高亮。
* 重构 `libs/audio_api` 的 encode 结果 marshalling 与 async job 结果管理，把 hex/bin 格式化、token/lyric line 拼接、timeline 拷贝和 follow 结果补建拆到独立实现文件，收敛 `bag_api` 内部职责，让同步 encode、异步 encode 和 follow-only 构建共用同一套结构化结果 contract。
* 优化 `libs/audio_core` 的 `flash` / `pro` 编码渲染路径，预先缓存 bit / symbol 模板波形并复用到整段 PCM 输出，降低长文本或多 segment 生成时重复三角函数计算的成本。

### 修复 (Fixed)
* 修复 encode follow 数据里的标点归属契约，确保 `HELLO, WORLD!`、`「神机」，启动。` 这类文本会把标点作为独立 token 暴露，但歌词行文本仍保持原始显示顺序，避免前端同时做 token 卡片与歌词排版时出现边界不一致。
* 修复 `WBAG` metadata 序列化/解析对 segment 边界的校验，要求 `segment_sample_counts` 数量与 `segment_count` 一致且总和覆盖 `pcm_sample_count`，避免长文本分段信息落盘后与真实音频采样范围脱节。
